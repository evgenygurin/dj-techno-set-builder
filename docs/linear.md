# Linear ↔ GitHub/GitLab — как гарантированно линковать задачи, ветки, PR/MR и коммиты

> Цель: чтобы **каждый PR/MR и коммит однозначно связывался с задачей Linear**, а статусы обновлялись предсказуемо.

## TL;DR (правило для этого репо)

1) **Ветка** должна начинаться с Linear issue ID: `BPM-123-...`  
2) **PR title** должен начинаться с Linear issue ID: `BPM-123: ...` (не в скобках и не в конце)  
3) **PR/MR description** должен содержать строку линковки (magic word), например:
   - `Fixes BPM-123` — связывает и (при включённой автоматизации) двигает статус/закрывает по merge
   - `Related to BPM-123` — только связывает (без “закрытия”)

Если PR уже создан с неправильным заголовком — исправить можно **редактированием title** или добавлением magic word в описание (см. ниже).

В этом репозитории формат заголовка PR дополнительно проверяется GitHub Actions workflow: `.github/workflows/pr-title.yml`.

---

## Как Linear связывает GitHub PR с задачей

### Авто-линковка (рекомендуется)

Linear связывает pull request с задачей, если **issue ID присутствует в названии ветки или в PR title** *до создания PR*.  
Поэтому самое надёжное — всегда начинать ветку и PR title с `TEAM-123`.  

### Линковка “после факта” и несколько задач на один PR

Если PR уже создан, или нужно связать **несколько** задач с одним PR, используйте **magic words** вместе с issue ID в **PR description**.

Важно:
- Для GitHub линковка через magic words работает из **PR description** (не из комментария).
- Если используете magic words, и задача не назначена ни на кого, Linear может **автоматически назначить её на автора** magic word (как на ответственного).

## Magic words (Linear)

Формат:
- `<magic word> <ISSUE-ID>` — пример: `Fixes BPM-123`
- `<magic word> <ISSUE-URL>` — если удобнее ссылкой

### Closing (закрывающие)

Эти слова **закрывают** задачу при merge в default branch (если включена автоматизация Git-интеграции):

| Magic word | Когда использовать |
|-----------|-------------------|
| `Fixes` / `Fix` | Исправление бага |
| `Closes` / `Close` | Закрыть задачу как выполненную |
| `Completes` / `Complete` | Завершение работ |
| `Resolves` / `Resolve` | Решение проблемы/инцидента |
| `Cancels` / `Cancel` | Отмена задачи |

### Non-closing (незакрывающие)

Эти слова **только связывают** PR/MR с задачей, без автоматического “Done”:

| Magic word | Пример сценария |
|-----------|------------------|
| `Related to` | Контекст/частичная работа |
| `Duplicate of` | Дубликат |
| `Blocks` / `Blocked by` | Зависимости задач |
| `Depends on` / `Dependent on` | Зависимости задач |
| `Splits from` / `Split into` | Декомпозиция |
| `Merges into` / `Merged from` | Объединение задач |
| `Contributes to` | Вклад в большую задачу без закрытия |

## Автоматическое движение статусов (GitHub integration)

Если включить **Issue status automation** в настройках GitHub-интеграции Linear:
- при **push ветки** с issue ID задача автоматически переходит в **In Progress**
- при **merge PR в default branch** задача переходит в **Done**

Эти правила можно ограничивать по веткам (branch-specific rules), чтобы, например, “технические” ветки не двигали статусы.

---

## Commit linking

Linear может показывать коммиты в activity feed задачи, если **в commit title или description есть issue ID** (`BPM-123`).

Рекомендация: даже если ветка/PR уже связаны, полезно включать issue ID и в ключевых коммитах — проще искать историю.

## Как исправить PR/MR, если заголовок неверный (как на скриншоте)

### GitHub PR

1) Отредактировать title PR и привести к формату: `BPM-123: <краткое описание>`.
2) В описание PR добавить одну строку:
   - `Fixes BPM-123` — если PR должен закрыть задачу
   - `Related to BPM-123` — если PR не должен закрывать задачу автоматически

### GitLab MR

GitLab-интеграция Linear имеет ограничения:
- надёжная линковка делается через **MR title** (issue ID должен быть в заголовке до открытия MR)
- для нескольких задач используйте magic words **в MR description**
- линковка через **комментарии** и **commit messages** для GitLab-интеграции не поддерживается

---

## Рекомендации “без серых зон”

- **Один PR = одна задача**. Если нужно иначе — закрывающая magic word только для одной задачи, остальные — `Related to` / `Contributes to`.
- **Issue ID — первым токеном** в ветке и PR title: это надёжно и для Linear, и для любых локальных regexp/автоматизаций.
- Не прячьте issue ID в скобках или в конце заголовка — это хуже читается и чаще ломает парсеры.
